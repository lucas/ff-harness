## Decision log

Running list of design decisions made during the build. Each entry is one line plus a `Why:` line. Append as new non-obvious decisions are made.

- **2026-06-13 â€” Two-worker stage-mapped setup chosen** (DeepSeek chat / Qwen code) over a single worker. **Why:** split is free for code-gen (Qwen3-Coder:free at 200 req/day), keeps the $1 spend headroom, and makes the rubric Worker-pillar bonus permanently on without any extra ceremony.
- **2026-06-13 â€” `html5lib` + `tinycss2` for site validation** chosen over external CLI linters. **Why:** pure-Python so no shell dependency, runs inside the orchestrator process, and JS validation is explicitly deferred in v1 (no good pure-Python option).
- **2026-06-13 â€” Bottom-up build order** (data â†’ service â†’ API â†’ frontend). **Why:** each layer is testable in isolation before the next is started; upper layers consume stable interfaces; MockWorker carries the orchestrator through Step 7 so the loop is provably correct before any LLM call.
- **2026-06-13 â€” UUID7 (TEXT) for all `id` columns.** **Why:** time-ordered (first 48 bits are ms timestamp) so ordering by `id` gives chronological order, no `AUTOINCREMENT`, single `new_id()` helper everywhere, clean.
- **2026-06-13 â€” FK columns on `events` (`material_id`, `checkpoint_id`, `alarm_id`) and a back-link on `alarms` (`triggered_by_event_id`, unenforced).** **Why:** joining `events` to any of the three referenced tables gives a queryable trace; the `alarmsâ†’events` back-pointer would create a cycle with the FK from `events.alarm_id`, so it's a logical reference only.
- **2026-06-13 â€” 429 â†’ auto-swap to fallback paid model, log `is_fallback=1`, append `model_swapped` event.** **Why:** keeps the demo running without manual intervention; makes the swap visible in the UI and queryable in `spend_log`; bounded blast radius (single retry, then `tool_failed` alarm).
- **2026-06-13 â€” Chat-first UI redesign â€” conversation derived from events (`human_resumed`/`worker_output`), input area context-sensitive on `session.status` + pending materials, details accordion, per-type event summaries instead of JSON expanders.** **Why:** the prior page front-loaded a debug timeline; users want to see what the agent said and what they need to answer next, not a wall of JSON. Pure projection functions in `harness/api/view_helpers.py` keep Jinja logic-free and unit-testable.
- **2026-06-13 â€” `iter_since_approval` resets on ANY `human_resumed` (bug fix) + `continuation_approval` pending material added when iter/spend caps trip.** **Why:** the old code only reset the counter when the user approved a brief or mockup â€” a freeform `ask_user` answer left the counter ticking and the user could hit the cap immediately after answering. Resetting on ANY resume matches the intent ("autonomous iterations between human inputs"). The new `continuation_approval` kind gives the UI something concrete to render when a cap trips, rather than a paused-but-empty input area.
- **2026-06-13 â€” State-based alarms (`iteration_limit_reached`, `spend_cap_reached`) auto-resolve when their condition no longer holds.** **Why:** these alarms describe a current state (counter vs cap, spend vs cap), not a past event. Leaving them `resolved=0` forever after the user approves a continuation or the rolling-24h spend window rolls is misleading. The orchestrator now calls `_resolve_obsolete_state_alarms` at the top of each loop iteration. Event-based alarms (`tool_failed`, `output_schema_violation`) are intentionally excluded â€” they reflect a past incident and are append-only history.
- **2026-06-13 â€” `POST /sessions/{id}/resume` auto-unsticks awaiting_human sessions via `orchestrator.force_continue`.** **Why:** the previous `/resume` route was a no-op when status was `awaiting_human` because `run_until_pause` exits early on non-`active` sessions. The user's mental model is "/resume means continue past whatever is blocking me", so the route now calls `force_continue` first: auto-approve any pending `continuation_approval` material (recorded as `auto_approved_via_resume: True` for observability), reset the iter counter, flip status back to `active`. Real content gates (freeform `ask_user`, subject-based `approval`) are deliberately left pending â€” those still require `/answer`. Reuses the existing `human_resumed` event + `user_approval` material types; no new schema.
- **2026-06-13 â€” `request_approval` bubbles render as subject-aware cards (no JSON to end users).** **Why:** the prior UX dumped the `details` dict as a JSON blob inside a collapsible `<details>` â€” fine for an engineer, jarring for the non-technical small-business owner the demo is built for. New `_render_approval_card(subject, details)` helper in `harness/api/view_helpers.py` returns `(body_html, body_plain)` directly (bypasses mistune; every value `html.escape`d before concatenation). `business_brief` gets hoisted name + tagline headings and a `<dl class="brief-rows">` grid (hours dict â†’ `Monâ€“Thu` style day labels; palette â†’ small inline `.swatch` colour chips behind a strict `^#[0-9A-Fa-f]{3,8}$` regex guard so non-hex values can't poison `style="background:â€¦"`; contact sub-dict â†’ phone/email/address hoisted to top-level rows). `mockup` renders a compact heading + "Approve the layout above" prompt (the ASCII lives in the prior `render_mockup` bubble). Unknown subjects fall back to a title-cased heading + labeled list â€” new subjects work without code changes. The `_tool_call_body` contract was extended with a `meta["body_html"]` override so a tool can ship its own pre-rendered HTML; `build_conversation` pops it before handing meta to the template. Tests cover the brief layout, palette swatch + invalid-value rejection, hours/contact/socials specials, mockup compaction, generic fallback, XSS escaping, and an end-to-end web-UI render assertion.
- **2026-06-13 â€” `render_mockup` upgraded to themed HTML + sandboxed iframe preview alongside the existing ASCII output.** **Why:** the ASCII wireframe was great for the deterministic `mockup_renders` checkpoint and for screen readers, but the chat panel just showed `Rendered mockup (N sections)` â€” the user had no visual sense of what the agent had laid out. New `_build_mockup_html` in `harness/services/tools/mockup.py` emits a self-contained `<!doctype html>` document (no remote fonts / CSS / JS / images) themed off the latest `business_brief` material (name + tagline as the hero heading; `palette.primary` / `palette.secondary` as the header band + hero tint â€” gated by the same `^#[0-9A-Fa-f]{3,8}$` regex used in the approval-card palette renderer so non-hex values fall back to neutral defaults and can't poison inline styles). The persisted `mockup` material content grew two fields (`html`, `themed`) but kept `ascii` + `regions` byte-identical so the checkpoint, events log, and screen-reader paths are unchanged. View-helper renders the HTML inside a maximally-restrictive `<iframe sandbox="" srcdoc="...">` in the chat bubble; `html.escape(doc, quote=True)` guards the srcdoc attribute boundary. `_tool_call_body` now takes `materials_by_id` so it can look up the latest mockup material and pull the html out. New `store.latest_material_by_type(session_conn, type)` helper (single query, `ORDER BY id DESC LIMIT 1`, leveraging UUID7's time-ordering) is the building block for any "find the latest X" lookup. Twelve new tests cover: result/material shape, themed vs. neutral output, palette XSS rejection, brief-name HTML escaping, ASCII determinism across brief states, iframe rendering in the conversation projection, srcdoc attribute escaping, fallback-to-plain when no material is on hand, and an end-to-end web-UI render.
- **2026-06-13 â€” Rewind feature: restart any session from a prior `awaiting_human` event (destructive but tracked).** **Why:** demo storytelling and recovery from a bad answer both benefit from being able to roll back. UUID7 ids are time-ordered so "after a given event" is a simple `WHERE id > target_event_id` predicate â€” no per-table timestamp comparisons, no parallel index. The truncation runs in a single transaction (events â†’ alarms â†’ checkpoints â†’ material, FK-safe because events references the other three) so partial state is impossible; an induced failure rolls everything back. The original pending material is re-pended and a `rewound` audit event with the row counts is appended so the action is visible in both the event log and (as a chat divider) the chat panel. `spend_log` is intentionally untouched: it represents real cost, not session state â€” rewinding shouldn't pretend the money wasn't spent. Layer rule preserved (store â†’ orchestrator â†’ route â†’ view-helpers â†’ template). New route `POST /sessions/{id}/rewind`, new `EventType.REWOUND`, new `store.rewind_to_awaiting_human` + `orchestrator.rewind_session`, new UI affordance (small subtle button on each `awaiting_human` row gated by a `confirm()` dialog), new `.chat-divider` CSS + projection role `divider`. Tests added across all five layers; gate command still green.
- **2026-06-13 â€” Mockup theming bug: drop auto-seeded `RESTAURANT_SEED_BRIEF` on session creation; persist approved `request_approval(subject='business_brief')` details as a `business_brief` material at approval time.** **Why:** the two bugs combined caused `render_mockup` to render "Maria's Pizzeria" for every real user. POST `/sessions` was eagerly persisting the seed brief "for fast demos" and the orchestrator never wrote the user's actual brief on approval, so the seed was the only `business_brief` material on file. Now POST `/sessions` creates an empty session, and the orchestrator's resume-fold step (`subject == "business_brief"` branch in `_consume_resume_if_present`) walks back to the pending material the approval responded to (`_pending_for_resume`), extracts `details`, and persists a fresh `business_brief` material before evaluating the checkpoint. Empty / missing `details` is a defensive no-op. `RESTAURANT_SEED_BRIEF` stays defined for opt-in demos / tests. No new event types, no new material types; layer rule preserved.
- **2026-06-13 â€” Chat UI polish: server-side markdown via `mistune`, de-duped input area, "Otherâ€¦" keeps options visible.** **Why:** three small fixes that together close the gap between the chat panel's intent and its actual feel. (1) The old design emitted raw markdown (`**bold**`, `*` bullets) into chat bubbles; agents commonly produce markdown so it needs to render. `mistune` is the smallest reasonable pure-Python markdown library, fast, and supports `escape=True` which is essential because LLM output is untrusted â€” without it a model that returned `<script>...</script>` would inject JS into the page. A single module-level renderer is built once at import (no per-request setup cost). Only agent bubbles whose body comes from the LLM (`final.summary`, `escalate.reason`, `ask_user.question`, `request_approval.subject`) go through the renderer; internal tool-call summary strings we constructed (e.g. `"Wrote index.html (1234 bytes)"`) and all user bubbles are HTML-escaped plain text â€” they're already formatted. Each conversation entry now carries both `body` (raw) and `body_html` (safe HTML the template emits). (2) The input area used to restate the pending question/subject/details that was already in the chat bubble immediately above â€” duplicate noise. Dropped the restatement; the input area now renders only the action controls plus a one-line muted cue (e.g. "Respond to the request above:"). The chat bubble above is the source of truth. (3) The "Otherâ€¦" button on options-based `ask_user` hid the option buttons when revealing the textarea â€” surprising. Now Otherâ€¦ opens the textarea BELOW the options-row and the buttons stay clickable; clicking a canned option still wins over typed text via the existing `submitter.dataset.option` path. No client-side JS for markdown â€” strictly server-side so the rendered HTML is auditable in tests.
