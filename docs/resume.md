## Settled architecture (no open decisions)
- **Engine:** durable, resumable, single-process staged-pipeline orchestrator; bounded loop.
- **Storage (two-tier):** core `harness.db` (users, sessions, spend_log) + per-session `sessions/{uuid}.db` (events, checkpoints, material, alarms, run_meta); append-only event log is the source of truth; `parent_session_id` for sub-agents.
- **Worker (decoupled):** swappable behind a data-only `Worker.act(WorkerContext)->WorkerResponse` protocol; stateless; **propose, don't dispose**; config/registry selection; contract tests; `MockWorker` reference.
- **Output contract:** strict JSON envelope `tool_call` / `final` / `escalate`, Pydantic-validated; validate→repair→alarm.
- **Guardrails:** tool allow-list, turn/token/time caps, **$1/day spend ceiling**, sandboxed file/git paths, **human approval every 10 iterations**.
- **Verification:** tiered (deterministic → independent verifier sub-agent → human) + harness self-testing (mock worker, replay/golden runs, fault-injection, property tests).
- **LLM:** OpenRouter, default `nvidia/nemotron-3-ultra-550b-a55b:free`, key in `.env`; configurable.
- **Also:** context compaction; sub-agents (own session); full event + cost logging.
- **Deploy:** FastAPI in Docker, host bind-mount for SQLite data, HTML/JS/CSS landing page = HITL + observability surface.

## v1 production fixes (post-Step 11)
- **`.env` loading:** `load_dotenv()` is called at import time of `harness/api/app.py` and inside `dependencies.build_default_app_context` (defense-in-depth). `OPENROUTER_API_KEY` and the four `MODEL_*` vars are now picked up from `.env` automatically.
- **Polling removed:** `session.html` no longer auto-polls every 2s (which was clobbering form state and `<details>` toggles). Replaced with an explicit "Refresh" link. The `?partial=1` route handler is retained for future smart-partial updates.
- **iter-cap continuation flow:** when the 10-iter human-approval gate or the $1/day spend cap trips, the orchestrator now persists a `pending_question` material with `content.kind = "continuation_approval"` so the UI has something concrete to render. `/answer` accepts the new kind and round-trips it back to the orchestrator, which distinguishes "Stop" (keeps the session paused) from "Approve" (resets the counter and resumes). The `iter_since_approval` counter resets on ANY `human_resumed` event, not just brief/mockup approvals.
- **Chat-first session UI:** the detail page is now a chat-style layout — agent and user bubbles projected from `worker_output` and `human_resumed` events, a context-sensitive bottom input area (Approve/Deny for approval, Approve/Stop for continuation_approval, options + Send for freeform `ask_user`), and a collapsed Details accordion housing the events table, checkpoints, active models, and the site iframe. Events use per-type human summaries (HH:MM:SS time + one-line text via `harness/api/view_helpers.format_event_for_table`) instead of raw JSON expanders. `awaiting.html` was removed; its responsibilities folded into `_session_main.html`.
- **State-based alarm auto-resolve:** `iteration_limit_reached` and `spend_cap_reached` are state-based — they describe a current condition, not a past event. The orchestrator now calls `_resolve_obsolete_state_alarms` at the top of each loop iteration (after resume-fold, before the per-turn guardrail check) and flips matching unresolved alarms to `resolved=1` when their condition no longer holds. `tool_failed` and `output_schema_violation` are event-based and never auto-resolved.
- **/resume unsticks awaiting_human sessions:** `POST /sessions/{id}/resume` now calls `orchestrator.force_continue(...)` BEFORE `run_until_pause` when the session is `awaiting_human`. `force_continue` auto-approves any pending `continuation_approval` materials (writes a `user_approval` with `auto_approved_via_resume=True`, appends a `human_resumed` event, marks the pending resolved), resets `iter_since_approval=0`, and flips status to `active` so the loop can advance. Freeform `ask_user` pendings and subject-based `approval` pendings (`business_brief`, `mockup`) are intentionally left pending — those are real content gates that still require `POST /sessions/{id}/answer`. Without this, `/resume` on a session at the iter/spend cap was a no-op because `run_until_pause` exits immediately when status isn't `active`.
- **Chat UI polish (markdown + de-duped input + Other-keeps-options):** agent bubble bodies are now markdown-rendered server-side via `mistune` (`escape=True` so embedded HTML in untrusted LLM output is escaped — XSS-safe; `hard_wrap=True` for chat-friendly newlines). The renderer is built once at module load in `harness/api/view_helpers.py` and reused per request. `build_conversation` decorates each entry with `body` (raw) and `body_html` (rendered or HTML-escaped). The input area no longer restates the pending question / approval subject / details — the chat bubble immediately above is the source of truth; only the action controls and a one-line muted cue remain. The "Other…" button on options-based `ask_user` no longer hides the option buttons — the textarea opens BELOW the options-row so the user can switch back to a canned option after seeing it (clicking an option still wins over typed text via `submitter.dataset.option`).
- **Approval bubbles render as subject-aware cards:** the old `request_approval` UX showed `Requesting approval: {subject}` with the `details` dict dumped as JSON inside a collapsible `<details>` — fine for debugging, terrible for non-technical end users. Replaced with a dedicated `_render_approval_card(subject, details)` helper in `harness/api/view_helpers.py` that returns `(body_html, body_plain)` directly (no mistune — every value is `html.escape`d before concatenation). `business_brief` renders a card with hoisted name + tagline, a `<dl class="brief-rows">` grid of known fields (industry, contact-hoisted phone/email/address, hours with `mon_thu → Mon–Thu` style labels, palette with small inline `.swatch` colour chips that only emit `style="background:#hex"` after a strict `^#[0-9A-Fa-f]{3,8}$` regex match, pages/service areas as comma-joined lists, socials as `Instagram: @x, …`). `mockup` renders a compact heading + "Approve the layout above" prompt (the ASCII art is already in the prior `render_mockup` bubble — no double-render). Any other subject falls back to a heading (title-cased subject) plus a labeled list of every top-level key (so a new subject type still surfaces cleanly without code changes). The `_tool_call_body` contract was extended so a tool can return a pre-rendered HTML override via `meta["body_html"]`; `build_conversation` pops it before passing meta to the template. The Jinja `_session_main.html` snippet that wrapped `details_json` in a `<details><pre>` was removed. CSS lives under `.chat-bubble` in `_base.html` so it can't leak into the page chrome. The system prompt nudges `request_approval` toward "structured objects like the Business Brief" so the model uses it (not `ask_user`) when the card UX is the right fit.
- **Rewind feature — restart any session from a prior `awaiting_human` event:** new `POST /sessions/{id}/rewind` route lets the user pick any past `awaiting_human` event in the Details accordion's events table and revert the session to that paused state. `store.rewind_to_awaiting_human(session_conn, target_event_id)` runs the truncation in a single transaction: validates the target event is `awaiting_human` (otherwise `ValueError` → 400), deletes events/alarms/checkpoints/material whose UUID7 id is greater than the target (FK-safe order: events first since they reference the other three), re-pends the material the session was paused on, and appends a `rewound` audit event with the row counts. `orchestrator.rewind_session(session_id, target_event_id, ...)` composes that with a core-DB session-row reset (`status='awaiting_human'`, `current_stage` taken from the target event row, `iter_since_approval=0`). The route does NOT call `run_until_pause` — the user re-answers via `/answer` which drives the loop. `spend_log` (core DB) is intentionally untouched; real cost stays on the books. UI: small subtle `class="rewind-btn"` button on every `awaiting_human` row in the events table, JS handler in `session.html` confirms via `confirm()` then POSTs `{target_event_id}` and reloads. `rewound` events render in the chat as a centered horizontal-rule-style `.chat-divider` (not a user/agent bubble) so the truncation moment is visible inline. New `EventType.REWOUND = "rewound"`; `format_event_for_table` recognizes the type with summary `"Rewound to {short_id} (removed N events)"` and highlight=True; `build_conversation` projects it to `role='divider'`. Tests cover store-layer truncation + transactional rollback on error, orchestrator session-row reset, full-loop continue-after-rewind, HTTP route 200/400/404 paths, view-helper divider/summary rendering, and end-to-end web-UI button presence + divider render. Layer rule preserved: store change in `services/store.py`, orchestrator change in `services/orchestrator.py`, route in `api/app.py`, projection in `api/view_helpers.py`, templates in `templates/`.
- **`render_mockup` now produces themed HTML + sandboxed iframe preview:** the tool persists both the existing `ascii` + `regions` payload AND a self-contained `<!doctype html>` themed wireframe plus a `themed: bool` flag in the `mockup` material content. When a `business_brief` material exists, `_build_mockup_html` (in `harness/services/tools/mockup.py`) themes the wireframe with the brief's name/tagline/primary_cta and the `palette.primary` / `palette.secondary` hex colors — gated by the same `^#[0-9A-Fa-f]{3,8}$` regex used elsewhere, so non-hex values fall back to neutral defaults (`#444` / `#eee`) and can't poison inline styles. Headers/heroes/footers get differentiated visual treatment (primary-color header band, secondary-tinted hero with brand + tagline + CTA, dark footer); everything else renders as dashed-border placeholder blocks. All brief-derived values are `html.escape`d at every insertion point. The HTML is fully self-contained (no remote fonts, no external CSS, no JS, no images) so it's safe to embed via iframe `srcdoc`. ASCII output is byte-identical to before — the `mockup_renders` checkpoint, events log, and screen-reader path all keep working. In the chat panel, `_tool_call_body` in `harness/api/view_helpers.py` looks up the most recent mockup material via `materials_by_id` and emits a `mockup-card` containing a caption (`Mockup preview ({N} sections[, themed])`) and a sandboxed iframe (`<iframe class="mockup-preview" sandbox="" srcdoc="..." loading="lazy" width="100%" height="420">`). `sandbox=""` is the most restrictive form — no scripts, no forms, no top-nav. The srcdoc value is HTML-attribute-escaped via `html.escape(quote=True)` so `"` characters inside the document can't break the attribute boundary. New `store.latest_material_by_type(session_conn, type)` helper supports the brief lookup; it orders by `id DESC LIMIT 1` (UUID7 is time-ordered).
- **`save_business_brief` worker-callable tool — defense in depth against the `ask_user`-for-approval path:** the orchestrator's resume-fold step persists a `business_brief` material when the user approves `request_approval(subject='business_brief')`. That covers one path. The agent often seeks brief sign-off via `ask_user(question='Does this look good?', options=['Looks good!', ...])` instead — that path never triggers the auto-persist and the user's actual brief never enters session memory. New `save_business_brief(brief: dict)` tool in `harness/services/tools/brief.py` lets the agent explicitly persist the brief from any approval path; it writes a `business_brief` material with `pending=False` via `store.persist_material` (same layer rule as `user.py` / `files.py` / `mockup.py` — stdlib + `harness.models` + `harness.services.store` only). Registered in `harness/services/tools/__init__.py::REGISTRY` and added to `harness/domain/website_builder.py::ALLOW_LIST` (7 tools now, not 6). The system prompt's Flow guidance step 1 instructs the agent to call `save_business_brief(brief={...full collected dict...})` IMMEDIATELY after the user has explicitly approved the brief (via 'Looks good!' / 'Yes' / 'Approved' / etc.) and BEFORE calling `request_approval(subject='business_brief', details=brief)` to formally close out the bootstrap stage. Example 1d in the prompt shows the exact envelope. Six new tests in `tests/services/test_tools.py` cover: happy-path persistence + material shape, missing-arg / non-dict-arg / empty-dict rejection (`error_kind=bad_args`), REGISTRY membership, and end-to-end (`render_mockup` themes its HTML against a brief persisted via `save_business_brief`). System prompt grew from ~5.7 kB to ~6.0 kB (under the 10 kB ceiling).
- **Mockup theming bug — actual user brief, not placeholder:** two related bugs together caused `render_mockup` to theme every real session with "Maria's Pizzeria" instead of the user's collected brief. (1) POST `/sessions` used to auto-seed `RESTAURANT_SEED_BRIEF` as a `business_brief` material on every new session — meant for "fast demos" but in practice it polluted real sessions because nothing replaced it. The auto-seed call in `harness/api/app.py` (`create_session`) was removed; the session is now created with no materials. `RESTAURANT_SEED_BRIEF` stays defined in `harness/domain/website_builder.py` for opt-in demos / tests. (2) When the user approved `request_approval(subject='business_brief', details={…collected brief…})`, the orchestrator evaluated the `business_brief_confirmed` checkpoint but never persisted `details` as a new `business_brief` material — so `_latest_business_brief` in `harness/services/tools/mockup.py` either found nothing (after fix 1) or the seed (before fix 1). The resume-fold step in `harness/services/orchestrator.py` (`_consume_resume_if_present`, `subject == "business_brief"` branch) now looks up the pending material that this approval responded to (via a new `_pending_for_resume` helper that walks back to the most recent prior `awaiting_human` event), pulls `details` out of its content, and persists a fresh `business_brief` material via `store.persist_material` BEFORE the checkpoint evaluation. Empty / missing `details` is a defensive no-op (no crash). One-shot persistence at approval time — not every turn. No new event types, no new material types; layer rule preserved (orchestrator + store changes stay in services, route change in api).

## Demo-domain flow (website builder)
onboarding (`bootstrap` skill) → **Business Brief** (defaults baked in; injected into every WorkerContext) → ASCII mockup approval → generate SEO-optimized HTML+CSS+JS + regenerate `sitemap.xml`/`robots.txt`/`llms.txt` each iteration → auto git commit → **final intent audit** vs the brief.
- **Tools:** `ask_user`, `request_approval`, `render_mockup`, `read`/`write`/`list` files, `validate_site`, `regenerate_seo_artifacts`, `git_history`/`git_revert`.
- **Named alarms:** iteration_limit_reached, spend_cap_reached, tool_failed, checkpoint_failed_repeatedly, output_schema_violation, verification_failed, verifier_low_confidence, verifier_disagreement, max_subagent_depth_exceeded, subagent_failed, compaction_stalled, intent_mismatch.
